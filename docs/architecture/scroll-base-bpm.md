# 基準速度（base BPM）怎麼決定

> 「同一個速度檔位，為什麼有的譜看起來特別快 / 特別慢？」的答案就在這裡。
> 實作：[`ManiaScroll.cs`](../../65/My%20project/Assets/Scripts/Sdo.Osu/ManiaScroll.cs) ·
> 測試：`Assets/Tests/EditMode/ManiaScrollTests.cs` · 語料掃描：[`tools/scan_base_bpm.py`](../../tools/scan_base_bpm.py)
> 設計面（SV 綠線、四層流速合成）在 [scroll-timing.md](scroll-timing.md)；這份只講 **base 怎麼挑**。

---

## 1. 流速公式（實際跑的）

```text
螢幕像素速度(t) = vBase × multiplier(t)

vBase          = anchorBpm × 速度檔位 × 1.6          (px/s，官方公式，見 memory sdo-note-scroll-speed)
multiplier(t)  = SV(t) × baseBeatLength / 目前段落的 beatLength
note 的位置     = judgeLineY + vBase × ∫ multiplier dτ   (osu SequentialScrollAlgorithm)
```

| 名詞 | 值 | 誰決定 |
|------|----|--------|
| `anchorBpm` | 預設 **固定 130**（`ManiaScroll.DefaultReferenceBpm`）；`scrollFollowsSongBpm=true` 時改用 base BPM | `ScreenGameplay.referenceBpm` |
| 速度檔位 | 房間「速度」（`RoomConfig.speedSteps`） | 玩家 |
| `baseBeatLength` | **本文主題**：譜面的「基準速度」 | 演算法（§2） |
| `SV(t)` | osu 綠線（.gn 沒有；NX/online frame_type 33 走另一條 `ScrollSpeeds`） | 譜面 |

重點：**預設模式下 `anchorBpm` 是固定的 130，所以 base BPM 不影響整體快慢的絕對值**——它決定的是
「譜面裡哪一段算 1.0×」。base 挑錯 → 該段以外的部分全部被乘上一個常數倍率。

## 2. 挑 base 的規則

底子是 osu 的 `Beatmap.GetMostCommonBeatLength`，加上兩個我們自己的修正：

**步驟 1（osu 原樣）——時間加權。**
只看 uninherited timing point（真正的 BPM 段；綠線 SV 不算）。每段的權重 = 它持續的毫秒數
（這一點 → 下一個 BPM 點；最後一段 → 最後一顆 note）。第一段一律從**時間 0** 起算
（osu 原始碼 `i == 0 ? 0 : t.Time`）。

**步驟 2（修正 A）——±3% 併群。**
相對誤差 ≤ `TempoClusterTolerance = 3%` 的 beat length 視為**同一個速度**。
（osu 是逐一 exact match，抖動 BPM 會把主速度切碎——見 §5 的 5028。）

**步驟 3（修正 B，非對稱）——選「站得住腳的最快的群」，不是時間最多的群。**

```text
合格(群) = ( 最長連續 ≥ FastTempoMinRunMs(6s)  或  累計時長 ≥ FastTempoMinTimeFraction(20%) )
           且  該群的 note 數 ≥ 全譜 note 的 FastTempoMinNoteFraction(5%)

base 群  = 合格群裡「最快」的那個，
           但不得超過「時間最多的群」的 MaxBaseTempoRatio(2.5) 倍；超過就往下一個較慢的合格群找，
           都沒有 → 退回時間最多的群（= osu 的答案）
```

**這條規則只會把 base 往「快」的方向調，永遠不會往慢的調。** 沒有合格的快群時就退回 osu 的答案，
所以最壞情況也只是「跟 osu 一樣」，不會比現況更糟。

**步驟 4（爭議譜）——用選歌畫面顯示的 BPM，再夾快段上限。**
當「最快合格群」≠「時間最多的群」時（＝長時間快段和長時間慢段各佔一塊，兩邊都站得住腳，
例：Electric Shock 125bpm 76% ／ 250bpm 24%），基準不是二選一，而是：

```text
base = clamp(表頭 BPM, 慢群, 快群)                       # 玩家在選歌畫面看到的數字
base = max(base, 快群 / FastSectionMaxMultiplier(1.5))   # 但快段不得超過 1.5×
```

表頭若正好落在中間（Still Alive 表頭 165、慢 110／快 220）就直接用表頭 → 快 1.33× ／ 慢 0.67×；
表頭若就是慢的那一群（多數情況）→ 往上抬到「快段剛好 1.5×」→ 慢段 0.75×。
**推論：快段 ≤ 1.5× 主速度的譜，base 就是主速度（＝ osu 的答案），完全不動。**

**步驟 5——代表值。**
每一群的代表值優先取**譜面表頭 BPM**（選歌畫面顯示的那個數字），表頭不在群內才退回群裡時間最長的成員。

> 全部常數都開在 `ManiaScroll` 上（`FastTempoMinRunMs` / `FastTempoMinTimeFraction` /
> `FastTempoMinNoteFraction` / `MaxBaseTempoRatio` / `FastSectionMaxMultiplier` /
> `TempoClusterTolerance`），每一個都有單元測試釘住語意。
> 要調就調這裡，然後 `python tools/scan_base_bpm.py` 重產 §6 的表。

## 3. 為什麼是「最快」而不是「最多」——非對稱的理由

使用者定奪（2026-07-21）：

> **就算 83bpm 佔 60%、167bpm 只佔 40%，還是必須用 167 當基準。**
> 快速的如果是細碎的短時間還能接受，只要有一段比較長時間的就不行；
> **因為快的人比較難反應，慢的人可以反應。**

這是一個**非對稱的可讀性事實**，不是「哪個比較有代表性」的統計問題：

- base 挑慢的 → 快段被乘 2.0×，note 擠成一團飛過去，**讀不到就是讀不到**，玩家當場爆。
- base 挑快的 → 慢段被乘 0.5×，note 變疏變慢，**看得清楚**，最多覺得拖；玩家還能自己把速度檔位往上調補回來。

錯誤成本不對稱 → 決策就該偏向快的那邊。門檻的作用只是排除「細碎短爆發」（1～2 秒的 gimmick 維持
>1× 是可接受的，也是譜面刻意的表現）。

### 但「兩邊都很長」的譜要折衷（使用者補充，2026-07-21）

> Electric Shock 125→250、Monster 128→256 **這種應該要取一個中間的速度**，讓快的部分不會太快、
> 慢的部分也不會太慢，因為兩段確實都佔據了一定時間。
> 這種有爭議的（長時間快速 + 長時間慢速）**就用外面 BPM 顯示的當作基準**。

於是有了 §2 步驟 4：爭議譜的基準＝**選歌畫面顯示的 BPM**（玩家調速度檔位時看的就是這個數字），
再加上「快段最多 1.5×」的夾擠——因為實測 25 首爭議譜裡有 **15 首的表頭寫的就是慢的那一群**，
照抄表頭快段又會回到 2.0×（＝ 5028 原本的病）。夾完之後：快 1.5× ／ 慢 0.75×，兩邊都不極端。

### 業界怎麼做（查證過的前例）

| 遊戲 / 做法 | 內容 | 對我們的意義 |
|---|---|---|
| **StepMania MMod**（`m550`） | 依歌曲 **DISPLAYBPM 的最大值**換算 xMod：「若歌曲 150→300bpm，m600 就是 300–600 的範圍」——**用最快的當基準**，慢段自然變慢 | 與使用者的直覺完全一致；「以快為準」本來就是業界的一種標準速度模式 |
| **StepMania `#DISPLAYBPM`** | 譜面作者手動宣告顯示 / 計算用的 BPM（或範圍），常搭配 scroll rate 補償自然的漸慢 | 前例：自動規則之外要留**每首手動覆寫**的後路（我們還沒有，見 §7） |
| **osu! issue #10267**（mania scroll 不該跟 BPM 變） | 玩家把 scroll 調到「自己能讀的最快」，BPM 一變就破功；mapper 只好用 SV 扳回來，「劇烈的 BPM 變化否則不可玩」 | 佐證：**變快才是致命的**；社群的解法方向也是不要讓 BPM 拉高 scroll |
| **Quaver `BPMDoesNotAffectScrollVelocity`** | 直接讓 BPM 不影響 SV，並提供把 SV 正規化到 common BPM 的工具 | 等同我們的「歌曲變速」關（`constantScroll`）；但正規化的錨點仍是 common BPM，「錨點怎麼挑」躲不掉 |
| **osu! issue #11446**（most-used BPM is incorrect） | 連 osu 自己的 most-common BPM 在多 BPM 譜上都常給出令人意外的值 | 佐證：直接照抄 most-common 不是安全牌 |

## 4. 「長時間」到底怎麼算

目前採用「**最長連續 ≥ 6 秒** 或 **累計 ≥ 20%**，且**該群 note 佔比 ≥ 5%**」。理由：

- **主判準用「最長連續」**：玩家問的是「這段夠不夠久到我必須改用它的速度讀譜」。
  6 秒 ≈ 8 小節 @160bpm ≈ 16 小節 @320bpm，遠超過「一個過門」的長度。
  一段 2 秒的 ×2 爆發撐不到 6 秒 → 不當基準（有測試釘住）。
- **累計 20% 補網**：有些譜把快段切成 3～4 塊散在各處（每塊 4～5 秒），只看連續會漏掉，但總量已達全曲 1/4。
  實例：sdom1033 最長連續才 4.4 秒，但累計 23% → 仍然合格。
- **note 佔比 5% 防呆**：沒有 note 的快速間奏再長也不用讀，不該把整首拖慢。
  實例：sdom2140 有 77% 時間掛在 195bpm 但那段幾乎沒有 note → 它不會被當成「合格的快群」去搶 base
  （這首的 195 本來就是時間最多的群，所以 base 仍是 195，只是不是因為它快而被選上——見 §6 反例表）。
- **上限 2.5 倍**：防止「撐很久的 ×4 gimmick」把整首壓成 0.25×（爬行）。目前語料沒有任何一首撞到，純粹是保險絲。
  StepMania 的 MMod 就是沒有這條保險絲，才需要 DISPLAYBPM 手動救。

另外「快段上限 1.5×」也是一個門檻：它同時決定了**慢段會被拖到多慢**（÷2 的譜 → 0.75×）。
1.25× 會讓主體掉到 0.63×（太拖）、1.6× 則快段偏快，1.5 是使用者定的折衷點。

### 敏感度（`tools/scan_base_bpm.py --sensitivity`，462 首多 BPM 譜）

| 門檻 | 與 osu 原版不同的譜 |
|---|---|
| 連續 ≥ 4s 或累計 ≥ 15% | 19 首 |
| **連續 ≥ 6s 或累計 ≥ 20%（採用）** | **18 首** |
| 連續 ≥ 8s 或累計 ≥ 25% | 17 首 |
| 連續 ≥ 10s 或累計 ≥ 30% | 15 首 |
| 只看累計 ≥ 20%（不看連續） | 13 首 |
| 只看連續 ≥ 6s（不看累計） | 16 首 |

4～10 秒之間只差 4 首 → 這條規則對門檻**不敏感**，邊界上的譜就那幾首（都列在 §6）。
之後真的覺得哪首怪，查表調門檻即可。

### 考慮過但沒採用的指標

| 指標 | 為什麼沒用 |
|---|---|
| 「快段的 note 密度（note/秒）」 | 更貼近真實難度，但要再多兩個門檻，且與「連續秒數」高度相關；留作 v2 |
| 「以小節數（8 小節）算而不是秒數」 | 小節數在慢歌會變得很長（8 小節 @90bpm = 21 秒），失去「玩家要花幾秒適應」的物理意義 |
| 「一律取全譜最快的 BPM」（純 MMod） | 一顆 1/4 拍的 999bpm gimmick 就毀掉整首；MMod 也得靠 DISPLAYBPM 收拾 |
| 「一律取表頭 BPM」（不分爭議與否） | 弄壞 7 首反向案例（sdom1186 表頭 65／主體 130、sdom2006 68/136、sdom0215 111/222…），那些譜取主體才對。所以表頭只在**爭議譜**當基準，而且夾在兩群之間 |
| 「爭議譜取兩群的（加權）幾何平均」 | 也是合理的中間值（Electric Shock → 147～177），但玩家不知道那個數字從哪來；用選歌畫面顯示的 BPM 當基準對玩家可解釋得多 |
| 「×2 / ÷2 折疊成同一族再選代表」 | 折疊後分不出「主速度＋半速段」與「半速記譜＋主體加倍」，實測 12 首裡有 7 首往錯的方向改 |

## 5. 起因個案：sdom5028《My Dearest》

hard 難度、`sdom5028K.gn`、表頭 BPM 169.0、最後一顆 note 90.4 s：

| beat length | BPM | 總時長 | 佔比 |
|---|---|---|---|
| 714.711 ms | **83.95** | 34.31 s | 38.0 % ← 單一最大塊 |
| 357.356 ms | 167.90 | 22.07 s | 24.4 % |
| 355.030 ms | 169.00 | 10.30 s | 11.4 % |
| 356.718 ms | 168.20 | 8.92 s | 9.9 % |
| 1429.252 ms | 41.98 | 5.72 s | 6.3 % |
| 361.446 / 366.972 / 352.941 / 359.281 ms | 166 / 163.5 / 170 / 167 | 共 5.67 s | 6.3 % |
| 其他（開頭 ritard：160/140/120/100/90/75/68） | — | 共 1.66 s | 1.9 % |

- **舊行為**：exact match 比大小 → 84bpm 以 38% 勝出當 base → 佔全曲一半以上的 168 家族全部變 **2.0×**
  （「大部分時間都在飛」）。
- **修正 A（併群）**：163.5–170 併成一群 = 46.5 s（51%、最長連續 15.3 s、393/641 顆 note），
  勝過 84 的 34.31 s；代表值取表頭 **169**。
- **修正 B（非對稱 / 爭議譜）**：這首的 168 群同時是「時間最多」也是「最快合格」，所以不算爭議譜，
  基準就是它、不需要折衷。非對稱規則與表頭折衷是為了 §6 那些「慢的比較多、但快段也很長」的譜。

結果：快段 1.0×、中段半速 0.5×、41.98 收尾段 0.25×。

`python tools/scan_base_bpm.py --gn sdom5028k` 可以隨時複驗。

## 7. 還沒做的 / 日後可能要動的

1. **每首手動覆寫（`baseBpm`）**——目前**沒有**。StepMania 用 `#DISPLAYBPM` 解決自動規則的例外；
   我們如果之後遇到「規則怎麼調都不對」的單一譜面，最省事的做法是在
   `StreamingAssets/song_name_overrides.json` 每首歌的物件裡加一個 `baseBpm` 欄位
   （寫入器 `SongOverridesWriter` 已經是外科手術式編輯，加欄位不難），由譜面編輯器存檔。
2. **玩家端開關**——現在只有「歌曲變速」（`constantScroll`，等同 osu Constant Speed / Quaver 的
   BPM-不影響-SV）。若之後想給玩家 StepMania 那種 `Cmod / Mmod / xMod` 三選一，這份規則就是 Mmod 的底。
3. **難度別的 base**——base 是**逐難度**算的（timing point 來自該難度的 block），easy/normal/hard
   若 BPM 事件不同就會各自有 base。目前語料三個難度的 BPM 事件都一樣，尚未出事。
4. **note 密度指標（v2）**——見 §4。
5. **外部匯入譜（osu/StepMania）** 走同一條規則，但 `map.Bpm` 對 .osu 來說是「第一個 timing point」
   而不是表頭 BPM（.osu 沒有表頭 BPM），代表值那一步的意義略有不同——影響 ≤3%，暫不處理。
6. §6 的表要**跟著門檻一起更新**（`python tools/scan_base_bpm.py`）。
