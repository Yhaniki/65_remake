# DPS 格式說明（PAS00003）

本文件整理 **Super Dancer Online（SDO／超級舞者）** 系列中 **`.dps` 舞蹈／動作腳本** 的二進位格式中**已實測或可對照逆向**的部分，以及**依位元組順序排列、尚未解義**的區塊。

透過逆向 `sdo_stand_alone.exe.c` 中的 `FUN_00407a30` 與對 `12956.DPS` 的二進位分析，確認此格式具有**階層式結構**。

## 概述

- **魔術數字**：ASCII `PAS00003`（8 bytes）。
- **用途**：描述與 **GN 譜面** 搭配的動畫片段序列。
- **資料來源**：
  - **逆向分析**：`bak/test/sdo_stand_alone.exe.c` 中的 `FUN_00407a30`（針對 `PAS00003` 分支）。
  - **二進位校閱**：`12956.DPS`（位於 `Super Dance Online/Datas/DANCE/`）。

## 術語

| 名稱 | 說明 |
|------|------|
| **檔頭（Fixed Header）** | 魔術數字 + 型別 + 關聯 GN 路徑。 |
| **表頭延伸區（Header Tail）** | GN 字串結束後，到**主物件計數**開頭之前的未知位元組區塊。 |
| **動作項目（Item）** | 包含時間資訊（pre_a, pre_b）與其下屬的**子動作列表**。 |
| **子動作（Sub-item）** | 包含動畫檔名（*.mot）與執行期參數。 |

---

## 整體結構（邏輯階層）

```
┌──────────────────────────────────────────────┐
│  PAS00003 (8B) + type_u32 (4B) + GN path (\0) │
├──────────────────────────────────────────────┤
│  表頭延伸區（未知資料，長度不定）              │
├──────────────────────────────────────────────┤
│  主物件計數 (item_count, uint32)               │
├──────┬───────────────────────────────────────┤
│ 項目0 │ pre_a (開始 tick), pre_b (持續 tick),   │
│      │ sub_item_count (子動作計數)            │
│      ├───────────────────────────────────────┤
│      │ [子動作 0 ... 子動作 N]                │
├──────┼───────────────────────────────────────┤
│ 項目1 │ ...                                   │
└──────┴───────────────────────────────────────┘
```

## 1. 檔頭與表頭延伸區

| 偏移 | 大小 | 類型 | 欄位 | 說明 |
|------|------|------|------|------|
| 0x00 | 8 | ASCII | `magic` | 固定為 `PAS00003`。 |
| 0x08 | 4 | uint32 LE | `type_u32` | 實測多為 **4**。 |
| 0x0C | 變長 | C 字串 | `gn_path` | 關聯 GN 檔名，以 `\0` 結尾（後可能接填充 `\0` 至 2 字元對齊）。 |
| ...  | 變長 | bytes | `header_tail` | **未知區塊**。在實測樣本中常包含看起來像**記憶體殘值（指標）**的資料。 |

---

## 2. 項目計數與循環（Item Loop）

| 相對偏移 | 大小 | 類型 | 欄位 | 說明 |
|----------|------|------|------|------|
| (TBD)    | 4 | uint32 LE | `item_count` | 檔案中總共包含多少個時間項目。 |

### 2.1 單個時間項目（Item）

| 相對偏移 | 大小 | 類型 | 欄位 | 說明 |
|----------|------|------|------|------|
| +0 | 4 | uint32 LE | `pre_a` | **本段開始時間刻度**（tick）。 |
| +4 | 4 | uint32 LE | `pre_b` | **本段持續長度**（tick）。 |
| +8 | 4 | uint32 LE | `sub_item_count` | 下屬子動作的數量（對應舊版文件中的 `pre_c`）。多數為 **1**。 |

---

## 3. 子動作結構（Sub-item）

當 `sub_item_count` > 0 時，緊接其後。若 `sub_item_count = 1`，則總長度通常固定為 **305 bytes**（16 bytes 檔名 + 289 bytes mid）。

| 相對偏移 | 大小 | 類型 | 欄位 | 說明 |
|----------|------|------|------|------|
| +0 | 16 | bytes | `mot_name` | **ASCII `*.mot`**；`\0` 結尾並填充至 16 bytes。 |
| +16 | 289 | bytes | `mid` | **動作參數區**。包含大量 `uint32` 或 `float`。 |

**已知 `mid` 區塊參數（經 `random_dps_gen.py` 與生成模型比對確認，相對 `mid` 開頭移位）：**
- `+244`: `uint32 LE`，`current_frame`（當前段落動畫的起始幀）。
- `+248`: `uint32 LE`，`end_frame` （當前段落動畫的結束幀，計算方式為起始幀加上該動作分配拍數占比下所需的幀數）。
- `+252`: `float LE`，`duration_s`（當前段落動畫的持續秒數）。

**記憶體指標警告**：在 `mid` 區塊與 `header_tail` 中，時常觀察到形如 `A4 F1 12 00` 的位元組序列，這高度疑似是客戶端存檔時將**記憶體指標 (Pointer)** 直接寫入檔案的殘留物。這意味著該區塊大部分資料對離線解析可能並不具備穩定語義，編輯時建議沿用原始位元組。

---

## 4. 時間軸規則

- **連鎖**：`pre_a[i+1] = pre_a[i] + pre_b[i]`（時間鏈無縫）。
- **總時長**：`T_end = Σ pre_b`。
- **單位**：與 GN 譜面共用的離散 tick。

---

## 5. 工具支援 (`dps_edit.py`)

目前的 `dps_edit.py` 使用以 **317 bytes** 為步進的「動作列」模式（即 `12 bytes (pre_a,b,c) + 16 bytes (name) + 289 bytes (mid)`）。這在 `sub_item_count == 1` 的情況下完全相容。

- **`set-name`**：修改 `P+12..P+27`。
- **`set-duration`**：修改 `pre_b` (+4..+7) 並同步平移。

---

## 6. 與客戶端載入的對應（逆向摘要）

根據 `FUN_00407a30` 的邏輯：
1. 通過 `FUN_0040fdd0` 開啟檔案。
2. 檢查魔數。
3. 讀取 `type_u32`。
4. 調用 `FUN_0040f690` 讀取 GN 路徑。
5. 讀取 `item_count` 並開始循環。
6. 每個循環讀取 `pre_a`, `pre_b`, `sub_item_count`。
7. 根據 `sub_item_count` 調用內層循環：
   - 讀取 16 bytes `mot_name`。
   - 接著大量調用 `FUN_0040f690` 與 `FUN_0040f260` 填充物件屬性（形成 `mid` 區塊）。

## 8. 開場專用動作 (Intro Motions) 規則

透過對官方源始 `.DPS` (超過 1000 份樣本) 發起的排查腳本 (`get_intro_mots.py`)，確認動作軌跡呈現以下規律：
1. 音樂片段正式起舞前，必定會拉取一個特定的**開場引導準備動作** (約有 75 種變體，如 `wdance0375.mot`, `wnew3.mot`, `_006540.mot` 等)。
2. 這些「開場專用」的動作只應被放置在 `Item[0]`（第一行）。
3. 為了保證演出邏輯正常，如果在中間段落利用到這些特定開場標籤（Intro Motions），會產生突兀的不自然重置動畫，因此在隨機抽取池中必須設立**排他機制 (Exclusive Pool)**，避免其於後端被循環選出。
